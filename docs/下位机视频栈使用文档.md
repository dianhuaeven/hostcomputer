# 下位机视频栈使用文档

本文说明下位机视频栈的职责边界、日常操作、生命周期控制和排障方式。第一次部署只需要按《下位机视频栈快速启动》执行；改配置时看《下位机视频栈配置文档》。

## 1. 总体架构

```text
上位机
  | TCP/JSON
  v
host_bridge_node.py
  | 本机 video manager 控制口
  v
video_manager_node.py
  | Popen/terminate FFmpeg publisher
  v
MediaMTX
  | RTSP
  v
上位机拉流
```

职责边界：

- `host_bridge_node.py` 是上位机唯一 TCP/JSON 入口，监听 `0.0.0.0:9090`。
- `video_manager_node.py` 是视频生命周期 owner，监听 `127.0.0.1:18081`。
- `ffmpeg` 只作为每路摄像头的 publisher，由 `video_manager_node.py` 创建和回收。
- `MediaMTX` 是 RTSP server，监听 `0.0.0.0:8554`。
- 上位机不直接接触 FFmpeg，也不直接控制 MediaMTX。

## 2. 正式入口

所有日常操作使用：

```bash
cd /home/rera/hostcomputer/ros1_bridge
./video_stack.sh <command>
```

支持命令：

```bash
./video_stack.sh start
./video_stack.sh stop
./video_stack.sh restart
./video_stack.sh status
```

正式 ROS 模式启动：

```bash
ROS_SETUP=/home/rera/robot24_ws/devel/setup.bash ./video_stack.sh start
```

只验证 bridge 和视频协议，不发布 ROS 控制话题：

```bash
BRIDGE_MODE=--dry-run ./video_stack.sh start
```

如果还想避免启动真实 FFmpeg publisher：

```bash
BRIDGE_MODE=--dry-run VIDEO_MANAGER_DRY_RUN=1 ./video_stack.sh start
```

旧的 RTSP 脚本只作为底层调试兼容入口；日常部署不要单独手动启动 FFmpeg。

## 3. 生命周期状态

`./video_stack.sh status` 会显示：

```text
summary: online=5/5 desired=5 errors=0 mediamtx_listening=True
camera 0 paw_realsense online=True desired=True pid=9863 log=/tmp/rtsp_logs/paw_realsense.log error=
```

核心字段：

| 字段 | 含义 |
|------|------|
| `desired_online` / `desired` | 下位机期望这路流保持在线 |
| `online` | 当前 FFmpeg publisher 是否实际在线 |
| `pid` | 当前 FFmpeg 子进程 PID |
| `restart_count` | 这路流被重启的次数 |
| `last_exit_code` | 最近一次 FFmpeg 退出码 |
| `log_path` / `log` | 这路 FFmpeg 日志路径 |
| `last_error` / `error` | 最近一次错误 |

正常状态应为：

```text
online=5/5 desired=5 errors=0
```

生命周期由 `desired_online` 和 `online` 一起体现：

- `start`：`desired_online=True`，video manager 创建 FFmpeg 子进程，成功后 `online=True`。
- `stop`：`desired_online=False`，video manager 终止 FFmpeg 子进程，成功后 `online=False`。
- `restart`：先 stop 再 start，`restart_count` 增加。
- 意外退出：如果 `desired_online=True` 且 `online=False`，监控循环会按 backoff 自动重启。

## 4. 自动恢复逻辑

配置中 `enabled: true` 的视频源启动后会变成：

```text
desired_online=True
```

如果某路 FFmpeg 意外退出：

```text
online=False
desired_online=True
```

`video_manager_node.py` 会在监控循环中尝试自动拉起该路流。为了避免坏设备反复刷进程，失败后有短暂 backoff。

## 5. 手动控制单路视频

可以通过内部控制口临时控制某一路。下面示例控制 `camera_id=0`。

重启一路：

```bash
python3 - <<'PY'
import sys
sys.path.insert(0, "/home/rera/hostcomputer/ros1_bridge")
from video_manager_ipc import VideoManagerClient

c = VideoManagerClient("127.0.0.1", 18081, timeout=5)
print(c.stream_request(0, "restart"))
PY
```

停止一路：

```bash
python3 - <<'PY'
import sys
sys.path.insert(0, "/home/rera/hostcomputer/ros1_bridge")
from video_manager_ipc import VideoManagerClient

c = VideoManagerClient("127.0.0.1", 18081, timeout=5)
print(c.stream_request(0, "stop"))
PY
```

启动一路：

```bash
python3 - <<'PY'
import sys
sys.path.insert(0, "/home/rera/hostcomputer/ros1_bridge")
from video_manager_ipc import VideoManagerClient

c = VideoManagerClient("127.0.0.1", 18081, timeout=5)
print(c.stream_request(0, "start"))
PY
```

控制后查看状态：

```bash
./video_stack.sh status
```

## 6. 上位机请求如何转发

上位机只连接：

```text
host_bridge_node.py :9090
```

上位机发送 `camera_list_request` 时：

```text
host_bridge_node.py
  -> video_manager_node.py list_cameras
  -> camera_list_response
```

上位机发送 `camera_stream_request` 时：

```text
host_bridge_node.py
  -> video_manager_node.py start/stop/restart
  -> ack
  -> camera_info
```

请求中的 `params.action` 支持 `start`、`stop`、`restart`，`params.camera_id` 当前为 `0..4` 的整数。

bridge 不直接 `Popen(ffmpeg)`。

## 8. RTSP 验证

本机验证：

```bash
ffprobe -v error -rtsp_transport tcp \
  -show_entries stream=codec_name,width,height,r_frame_rate \
  -of default=noprint_wrappers=1 \
  rtsp://127.0.0.1:8554/frontcam
```

上位机验证时不要使用 `127.0.0.1`，应使用 `rtsp.public_host`：

```text
rtsp://192.168.1.83:8554/frontcam
```

## 9. systemd 部署

手动跑通后再启用 systemd。

启用前要确认 ROS master 会先启动，或者当前 `/etc/default/robot-video-stack` 里的 `ROS_MASTER_URI` 指向可达的 master；否则 `host_bridge_node.py` 在正式 ROS 模式下不会打开 `9090`。

```bash
cd /home/rera/hostcomputer
sudo cp deploy/systemd/robot-video-stack.service /etc/systemd/system/
sudo cp deploy/systemd/robot-video-stack.env.example /etc/default/robot-video-stack
sudo nano /etc/default/robot-video-stack
sudo systemctl daemon-reload
sudo systemctl enable --now robot-video-stack.service
```

确认 `/etc/default/robot-video-stack` 至少包含：

```bash
HOSTCOMPUTER_DIR=/home/rera/hostcomputer
VIDEO_SOURCES_CONFIG=/home/rera/hostcomputer/ros1_bridge/video_sources.local.yaml
ROS_SETUP=/home/rera/robot24_ws/devel/setup.bash
```

查看 systemd 状态：

```bash
systemctl status robot-video-stack mediamtx --no-pager
journalctl -u robot-video-stack --since '10 minutes ago' --no-pager
/home/rera/hostcomputer/ros1_bridge/video_stack.sh status
```

## 10. 常见问题

### `host_bridge_node` 不监听 `9090`

检查 ROS master：

```bash
ss -lntp | grep 11311
```

检查日志：

```bash
tail -200 /tmp/rtsp_logs/host_bridge_node.log
```

如果只是验证 bridge 协议、不接 ROS，也不启动真实 FFmpeg，可先用：

```bash
BRIDGE_MODE=--dry-run VIDEO_MANAGER_DRY_RUN=1 ./video_stack.sh start
```

### `video_manager_node` 正常，但某路 `online=False`

看该路 `error` 和日志：

```bash
./video_stack.sh status
tail -100 /tmp/rtsp_logs/<camera>.log
```

常见原因：

- `/dev/*cam` 不存在。
- 设备被其他进程占用。
- MediaMTX 未监听 `8554`。
- 摄像头不支持配置里的 `input_format` / 分辨率 / 帧率。

### `status` 出现旧的错误日志

`/tmp/rtsp_logs/*.log` 是追加写入的节点日志，可能包含历史启动记录。判断当前状态以 `summary` 和每路 `error` 为准。

清空历史节点日志：

```bash
./video_stack.sh stop
rm -f /tmp/rtsp_logs/video_manager_node.log /tmp/rtsp_logs/host_bridge_node.log
ROS_SETUP=/home/rera/robot24_ws/devel/setup.bash ./video_stack.sh start
```
